# PRD — Leon Feeding Tracker

## 1. Обзор

Личное mobile-first веб-приложение для фиксации истории искусственного кормления одного ребёнка (Леон, родился 25.04.2026). Стек: Next.js (текущая версия из `node_modules/next/dist/docs/`), MongoDB через Mongoose, shadcn/ui, Tailwind. Без аутентификации.

## 2. Доменные данные

### 2.1 Профиль ребёнка (фиксированный, сидится один раз)
- name: «Леон»
- birthDate: 2026-04-25
- birthWeightGrams: 3400
- feedingsPerDay: 8 (константа на этапе MVP)

### 2.2 Формула суточной нормы
- **Дни 1–10** (формула Зайцевой): `суточная_цель = 0.02 × вес_при_рождении × n`, где n — день жизни. Для Леона: `68 × n` мл.
- **С 11-го дня** — выбираемая формула из реестра `lib/planning/formulas.ts` (стратегии «зрелой стадии»):
  - `weight-fifth` (весовая, 1/5 массы): `суточная_цель = текущий_вес / 5` мл. Бесшовный стык 10↔11: 680 мл → при 3400 г 680 мл.
  - `rounded-plus-200` (округление + 200): `суточная_цель = round(текущий_вес / 100) × 10 + 200` мл. Пример: 3485 → 3500 → 350 → 550 мл; 4280 → 4300 → 430 → 630 мл. Стык 10↔11 не бесшовный.
- Активная формула задаётся `ACTIVE_MATURE_FORMULA_ID` в `lib/planning/formulas.ts`. Текущая активная — `rounded-plus-200`. UI-переключателя пока нет; будущий `baby.dailyFormulaId` подменит константу без изменения сигнатур.
- День жизни = `floor((сегодня − birthDate) / 24ч) + 1`, по календарным дням локального TZ.

### 2.3 Время и таймзоны
- БД хранит UTC.
- UI конвертирует в локальный TZ браузера (`Intl.DateTimeFormat`, `timeZone: undefined`).
- Кормление принадлежит дню по времени **начала**.
- Окно «дня» — 00:00–23:59 локального времени.

## 3. Модель данных

### 3.1 `baby` (1 документ)
```ts
{
  _id,
  name: string,            // "Леон"
  birthDate: Date,         // UTC
  birthWeightGrams: number,
  feedingsPerDay: number,  // 8
  timestamps
}
```

### 3.2 `weights` (история взвешиваний)
```ts
{
  _id,
  date: Date,              // UTC, к какому календарному дню относится взвешивание
  weightGrams: number,
  createdAt
}
```
Первая запись сидится: `date=2026-04-25, weightGrams=3400`.

«Текущий вес» = запись с максимальной `date ≤ сегодня`.

«Пора взвесить» = `(сегодня − max(weights.date)) ≥ 3 дня`.

### 3.3 `feedings`
```ts
{
  _id,
  startAt: Date,                    // UTC, обязательно
  endAt: Date | null,               // UTC, опц.
  volumeMl: number | null,          // опц.
  isTopUp: boolean,                 // false по умолчанию
  parentFeedingId: ObjectId | null, // null для основного, заполнен для докорма
  timestamps
}
```

Валидации:
- `startAt` не в будущем.
- `endAt > startAt`, и `endAt − startAt ≤ 60 мин`.
- `0 ≤ volumeMl ≤ 200`.
- Запись можно создавать любым днём в прошлом.
- Перекрытия по времени не валидируются.

Удаление:
- Можно.
- При удалении основного у привязанных докормов `parentFeedingId = null` (становятся независимыми).

## 4. Алгоритмы плана

### 4.0 Базовые принципы (важно)

- **Любое следующее время (плановой метки или докорма) считается от `startAt` предыдущего кормления, не от `endAt`.** `endAt` нужен только для расчёта длительности (метрика истории) и валидации формы.
- **Стартовый план** дня — детерминированная функция от веса/даты/последнего кормления предыдущего дня. Не хранится, не сопоставляется с фактом.
- **Рабочий план** — динамически перестраивается после каждого факта. В ленте дня факты вытесняют затемнённые плановые метки.
- **Виртуальная метка докорма** — пункт ленты UI, не запись в БД. После тапа и сохранения становится `feeding {isTopUp:true, parentFeedingId}`.
- **Виртуальный докорм считается «обещанным» объёмом** в расчётах хвоста (COUNT-AS-PLANNED): он входит в `съедено_к_текущему_моменту` для целей пересчёта плана, но не в текстовое отображение «съедено» в шапке.
- Все времена локальные; сутки — `[00:00, 24:00)` локального TZ.

### 4.1 Цель суток для дня D

```
weight_for_D = weights, max(date) where date <= D  // fallback: birthWeightGrams
target_D    = day_of_life(D) <= 10
              ? 0.02 * birthWeightGrams * day_of_life(D)
              : MATURE_FORMULAS[ACTIVE_MATURE_FORMULA_ID](weight_for_D)
```

Активная стратегия зрелой стадии — см. §2.2. На текущий момент `rounded-plus-200`: `round(weight_for_D / 100) * 10 + 200`. Реестр в `lib/planning/formulas.ts`.

### 4.2 Стартовый план дня D

Параметры:
- `target = target_D` (см. 4.1)
- `prevFeed` = последний `feeding` (любого типа), у которого `startAt < 00:00 локального D`
- `anchor = prevFeed?.startAt ?? 00:00 локального D`
- Слоты строим: `slots = []`, `t = anchor + 3ч`, пока `t < 24:00 локального D`: `slots.push(t); t += 3ч`. Получаем `K_start` слотов.
- `portion_start = target / K_start` (а **не** `target / feedingsPerDay`). Стартовый план всегда покрывает суточную цель.
- Все слоты типа `kind: "planned"` с объёмом = `portion_start`.
- **Инвариант для стартового плана**: при шаге 3ч и сутках 24ч `K_start ∈ [7, 8]`. Минимум 7 — когда anchor = 00:00 D (нет кормлений в предыдущем дне). Максимум 8 — anchor в предыдущем дне. Случай `K_start == 0` для стартового плана математически недостижим. При изменении шага интервала или модели суток инвариант пересмотреть.
- **Замечание для рабочего плана** (4.3): `tailBefore` (длина хвоста до текущего факта) может стать 0, если предыдущий факт пришёлся в конец дня (например, 22:30, +3ч уже за полночь). В этом случае пайплайн 4.3.2 делает ранний выход: факт просто фиксируется, триггеры/толерантность/размазывание не запускаются (адаптировать в этом дне уже нечего). Стартовый план **следующего** дня D+1 при этом считается нормально — anchor для D+1 = startAt последнего кормления D, всё по §4.2.

Псевдокод:

```ts
function computeStartPlan(D: LocalDate, ctx: Ctx): Slot[] {
  const target = computeTarget(D, ctx);
  const prevFeed = ctx.lastFeedingBefore(startOfDay(D));
  const anchor = prevFeed?.startAt ?? startOfDay(D);
  const slots: Date[] = [];
  let t = addHours(anchor, 3);
  while (t < endOfDay(D)) { slots.push(t); t = addHours(t, 3); }
  if (slots.length === 0) return [];
  const portion = target / slots.length;
  return slots.map(time => ({ time, volumeMl: portion }));
}
```

### 4.3 Рабочий план (лента дня) — single next-slot shift (locked)

Используется на Today и в Day detail. Строится за один проход.

#### 4.3.1 Состояние, накапливаемое по фактам

Сортируем факты дня по `startAt`. Идём слева направо, поддерживая:
- `consumed` — сумма `volumeMl` всех уже обработанных фактов;
- `lastStart` — `startAt` последнего обработанного **основного** факта;
- `mainsToday` — число основных кормлений (`isTopUp:false`) в текущем локальном дне D;
- `lastShiftMin` — сдвиг для следующего слота, рассчитанный по последнему основному факту.

Исторические записи с `isTopUp:true` (старые данные) суммируются в `consumed`, но **не** двигают `lastStart`, не увеличивают `mainsToday` и не участвуют в расчёте сдвига. Новые записи с `isTopUp:true` больше не создаются.

#### 4.3.2 Single next-slot shift formula

`idealPortion = target / feedingsPerDay`.

Для основного факта `f` со ссылкой на предыдущий основной (`prev.startAt`):

```
dev = (f.volumeMl − idealPortion) / idealPortion

# v_shift — сдвиг по отклонению объёма (включительная зона ±0.2)
|dev| ≤ 0.2          → v_shift = 0
0.2 < |dev| < 0.6    → v_shift = round(30 · (|dev| − 0.2) / 0.4) · sign(dev)
|dev| ≥ 0.6          → v_shift = 30 · sign(dev)

# t_shift — сдвиг по раннему приходу, только если v_shift == 0
if v_shift == 0:
    earlyMin = ((prev.startAt + 3h) − f.startAt) в минутах
    earlyMin ≤ 30           → t_shift = 0
    30 < earlyMin < 60      → t_shift = round(30 · (earlyMin − 30) / 30)
    earlyMin ≥ 60           → t_shift = 30
    earlyMin < 0 (опоздание)→ t_shift = 0
else:
    t_shift = 0

shiftMin = v_shift + t_shift   # один из них всегда 0; результат в [-30, +30]
```

Если `mainsToday === 0` после всех фактов — сдвига нет (`lastShiftMin = 0`): у нас нет фактического объёма, чтобы посчитать `dev`. `prevDayAnchor` участвует только как `lastStart`-якорь для размещения первого слота.

#### 4.3.3 Построение хвоста плана

`lib/planning/tail.ts::buildTail`:
1. Первый слот хвоста: `firstSlot = lastStart + 3h + lastShiftMin`. Если `firstSlot ≥ 24:00 D` — слотов нет (cross-midnight drop).
2. Остальные слоты: спейсинг по `buildRhythmSlots`, заякоренный от `firstSlot` (см. §4.3.4 ниже).
3. Объём каждого слота: `idealPortion`. Без размазывания, без `kind`-дискриминатора, без режимных слотов.

```ts
type Slot = { time: Date; volumeMl: number };
```

#### 4.3.4 Ось расписания (когда) — `buildRhythmSlots`

- `remainingHours = (24:00 D − anchor)`. Если ≤ `FALLBACK_MIN_HOURS` (1ч) → слотов нет.
- Кандидаты K: `ceil(remainingHours/STEP_MAX) − 1 ≤ K ≤ floor(remainingHours/STEP_MIN) − 1`, где `STEP_MIN=2.5`, `STEP_MAX=3.5`.
- Если множество пусто → слотов нет.
- Из кандидатов выбираем K, минимизирующий `|alreadyToday + K − feedingsPerDay|`. При равенстве — шаг ближе к `STEP_IDEAL=3ч`.
- Шаг `step = remainingHours / (K+1)`. Слоты: `t_i = anchor + step·i` для `i=1..K`.

Якорь = `firstSlot` (см. 4.3.3); `alreadyToday` = `mainsToday + 1` (учитываем `firstSlot` как уже добавленный).

#### 4.3.5 Сборка ленты UI

```
timeline = sort_by_time(
   facts (kind: "feeding", style: bright)
   + tail.map(s => ({kind: "plan-slot", style: dim, ...s}))
)
```

Прошлые непокрытые плановые метки **не показываются** (хвост строится только в будущее от `lastStart + 3h + lastShiftMin`).

### 4.4 Триггер докорма — УДАЛЁН

Single next-slot shift заменяет дуальную ось «толерантность объёма + триггер докорма + размазывание». Новые записи с `isTopUp:true` больше не создаются: схема API форсирует `isTopUp:false, parentFeedingId:null` на POST/PATCH. Исторические top-up-записи в БД сохраняются и видны в `HistoryList` (метрика `topUpsCount`), но не влияют на план дня.

### 4.5 Сопоставление факта со стартовым планом — НЕТ

Подчёркиваем явно: факт не сопоставляется с конкретной плановой меткой. Стартовый план — отдельный read-only артефакт, рассчитанный от предыдущего дня. Все решения пайплайна 4.3.2 принимаются от **рабочей** `portionNow`.

### 4.6 Метрики истории дня

- **Факт суток** = Σ `volumeMl` за день (вкл. докормы, исключая записи с `volumeMl == null`).
- **Кол-во кормлений** = записей с `isTopUp=false` за день.
- **Кол-во докормов** = записей с `isTopUp=true` за день.
- **Средняя длительность** = среднее `(endAt − startAt)` по записям где `isTopUp=false` И `endAt != null`.
- **Недостача** = `target − факт_суток`. Положительная — недокорм, отрицательная — перекорм. Считается только для прошедших дней.

### 4.7 Worked example (4 мая, день 10)

Вход:
- `birthWeightGrams = 3400`, `target = 0.02 × 3400 × 10 = 680 мл`
- Последнее кормление 3 мая: `startAt = 22:52`, далее в 4 мая факты идут.

Стартовый план 4 мая:
- `anchor = 22:52` (3 мая). Метки: 01:52, 04:52, 07:52, 10:52, 13:52, 16:52, 19:52, 22:52 — `K_start = 8`.
- `portion_start = 680 / 8 = 85 мл`.

Факт 1: `01:23, volumeMl=75`.
- До факта: `consumedBefore=0`, `tailBefore=8`, `portionNow = 680/8 = 85`.
- `delta = 75 − 85 = −10`. `|delta| = 10 ≤ 8.5`? Нет (10 > 8.5). `delta < −17`? Нет (−10 > −17). → **2c, размазывание**.
- После: `consumed=75, lastStart=01:23`. Хвост: `t = 04:23, 07:23, 10:23, 13:23, 16:23, 19:23, 22:23` → 7 меток. `remaining = 680 − 75 = 605`. `605/7 ≈ 86.4 мл`. Лента: факт 75 + 7 затемнённых по 86 мл.

Факт 2: `02:50, volumeMl=30, isTopUp:true, parent=факт1` (докорм по факту, не виртуальный).
- При обработке: пересчитываем `consumed=105`. `lastStart` остаётся 01:23? **Нет, обновляем на 02:50.** Хвост заново: `t = 05:50, 08:50, 11:50, 14:50, 17:50, 20:50, 23:50` → 7 меток. `remaining = 680 − 105 = 575`. `575/7 ≈ 82.1 мл`.

…и так далее по каждому следующему факту. (Полная прогонка опускается — алгоритм детерминирован.)

Ключевой момент: после каждого факта **хвост перестраивается с нуля** от `lastStart`, минимизируя статус прошлого. Пайплайн 4.3.2 определяет лишь, как мы отреагируем (триггер/толерантность/размазывание), но фактический хвост всегда вычисляется одинаково (4.3.3).

## 5. Страницы (5 экранов, bottom tabs)

### 5.1 Today (`/`)
**Шапка**: цель суток (число), съедено / прогресс-бар, время следующего кормления + объём, день жизни, текущий вес, дата.

**Лента**: слияние фактов и хвоста плана в одной шкале времени. Тап на затемнённую плановую метку → форма (предзаполнена `time` метки и `portion`). Тап на горящий факт → редактирование.

**Коллапс**: «Идеальный план дня» — таблица 8 меток стартового плана.

**Баннер**: «Пора взвесить» (если `≥3 дня` от последнего взвешивания), dismissable; возвращается на следующий день, если не взвесили.

**FAB / кнопка**: «+ кормление» открывает bottom sheet.

### 5.2 History (`/history`)
Список дней по убыванию, виртуализация + подгрузка. Карточка: дата + день жизни, цель, факт, дельта (цвет), `N кормлений + M докормов`, средняя длительность.

### 5.3 Day detail (`/history/[date]`)
**Тот же компонент, что Today**, с параметром даты. Если дата прошлая — шапка показывает дельту вместо «следующего». Редактирование разрешено для любого дня.

### 5.4 Weight (`/weight`)
Список взвешиваний по убыванию. Кнопка «Добавить взвешивание» → форма (дата + вес в г). Бейдж в табах появляется при триггере «пора взвесить».

### 5.5 Settings (`/settings`)
Read-only данные ребёнка + редактируемое поле `feedingsPerDay`.

## 6. Форма кормления (bottom sheet, shadcn `Sheet`/`Drawer`)

Поля (все опциональны кроме `startAt`):

- **startAt**:
  - Chips: «Сейчас» (текущее, округление 5 мин), «−5», «−10», «−15», «По плану HH:MM» (если форма открыта тапом на плановую метку).
  - Fallback: shadcn time picker.
- **Длительность** (вычисляет `endAt = startAt + duration`):
  - Chips: 10 / 12 / 15 / 20 мин.
  - Fallback: ручной picker конца.
- **volumeMl**:
  - Stepper ±5 мл, дефолт = плановая порция.
  - Numeric keypad для произвольного значения.
- **isTopUp**: автоматически выставляется, если форма открыта от триггера докорма; иначе скрыто/false.

Кнопки: «Сохранить», «Удалить» (в режиме редактирования), «Отмена».
